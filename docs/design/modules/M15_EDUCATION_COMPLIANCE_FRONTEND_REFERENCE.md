## 1. Module Purpose

M15 helps school and college administrators prepare Nepal education-compliance records without treating exports as proof of government submission.

It provides iEMIS readiness, UGC/HEMIS-ready reporting, QAA readiness, validation errors, export history, and submission tracking by consuming authoritative data from M1 students, M4 academics, M7 HR, M11 finance, M12 notifications, and File Registry.

This module must not duplicate core student, staff, exam, accounting, file, notification, or audit truth.

## 2. Feature List

### Compliance Dashboard

**Purpose:** Shows readiness by compliance area and highlights missing or invalid data.

**Users:** School admin, college compliance officer, principal/campus chief.

**Frontend behavior:** KPI cards for readiness, report status, validation errors, and upcoming QAA/HEMIS tasks. All counts come from backend summary APIs.

**Backend alignment:** Backend owns official readiness state, validation count, and report run state.

### Institution Profile

**Purpose:** Captures legal, location, affiliation, tax, and accreditation information used across reports.

**Users:** Admin and authorized compliance officer.

**Frontend behavior:** Sectioned form with validation, evidence file upload through File Registry, and change audit timeline.

**Backend alignment:** Backend validates tenant, required fields, affiliation context, and sensitive field changes.

### Education Structure Mapping

**Purpose:** Maps SchoolOS classes, streams, programs, levels, faculties, years/semesters, and batches to reporting structures.

**Users:** Academic admin and compliance officer.

**Frontend behavior:** Hierarchical setup table with active/inactive status, missing mapping warnings, and import/export where backend supports it.

**Backend alignment:** Backend owns mapping rules and prevents mappings to inactive or unauthorized structures.

### iEMIS Readiness

**Purpose:** Shows school-level readiness for student/profile/government reporting data.

**Users:** School admin, principal.

**Frontend behavior:** Checklist grouped by institution, class/section, student, guardian, enrollment, and document completeness.

**Backend alignment:** M15 reads canonical M1/M4 data and reports readiness; it does not rewrite core student data silently.

### UGC / HEMIS Readiness

**Purpose:** Prepares college/campus data for UGC/HEMIS reporting and export.

**Users:** College compliance officer, campus chief.

**Frontend behavior:** Program/staff/student/scholarship summary tables, validation errors, report preview, export action, and submission status.

**Backend alignment:** Backend validates HEMIS-specific required fields and controls export generation.

### QAA Readiness

**Purpose:** Tracks accreditation status, expiry, evidence package, and renewal readiness.

**Users:** Campus chief, QAA coordinator, compliance officer.

**Frontend behavior:** QAA status card, expiry timeline, evidence checklist, protected file links, and readiness notes.

**Backend alignment:** Backend owns evidence metadata and protected file authorization.

### Report Generator

**Purpose:** Generates iEMIS, UGC/HEMIS, QAA, scholarship, staff, and custom compliance exports.

**Users:** Authorized compliance roles.

**Frontend behavior:** Select report type, academic/fiscal year, validation mode, preview, generate, download protected file, mark submitted.

**Backend alignment:** Backend generates exports server-side and stores files through File Registry.

### Submission History

**Purpose:** Keeps auditable history of generated, exported, submitted, rejected, and archived report runs.

**Users:** Compliance officer, principal/campus chief, auditor by permission.

**Frontend behavior:** Timeline/table with report type, status, generated by, exported file, submitted by, submitted date, rejection reason.

**Backend alignment:** Backend owns state transitions and audit trail.

## 3. Personas and Scope Boundaries

| Persona | Can access | Must not access |
|---|---|---|
| School admin | Institution profile, iEMIS readiness, report export where allowed | Other tenants, hidden platform notes, unrelated college-only reports |
| College compliance officer | HEMIS/QAA/staff/scholarship reports | Raw accounting journals, unauthorized student private notes |
| Principal/campus chief | Readiness summary, approval/submission history | Cross-tenant data, raw file keys |
| Accountant | Tax identity and scholarship financial summaries by permission | HEMIS mapping unless separately allowed |
| Parent/student | No M15 admin screens by default | Compliance exports, demographic aggregates, submission history |

## 5. Screen Specifications

### 5.1 Compliance Dashboard

**Purpose:** Give leadership a safe, backend-owned compliance readiness view.

**Layout:** Module header, readiness KPI strip, attention queue, upcoming deadlines, latest report runs, right drawer for selected validation group.

**Header actions:** Generate report, Open validation errors, Export summary.

**States:** Loading, empty, no results, permission denied, module locked, stale summary, validation service unavailable, report generation queued, export unavailable.

**Backend data needed:** Readiness summary, validation counts, report run status, as-of timestamp.

**Security:** No cross-tenant aggregates. No frontend-calculated compliance scores.

### 5.2 Institution Profile

**Purpose:** Maintain legal, location, affiliation, tax, and accreditation metadata.

**Layout:** Tabs for identity, location, affiliation, tax identity, accreditation, contacts, evidence files, audit.

**Header actions:** Save draft, Submit changes, View audit.

**Confirmations:** Sensitive fields such as PAN/VAT, affiliation, QAA status, and institution type require confirmation and audit reason when changed after reports exist.

**Backend data needed:** Current profile, validation schema, evidence file metadata, audit timeline.

**Protected files:** QAA certificates, affiliation letters, registration documents through File Registry.

### 5.3 Education Structure Mapping

**Purpose:** Map institution academic structures to reportable structures.

**Layout:** Tree/table showing university/board, faculty, level, program, year/semester, batch/section.

**Header actions:** Add program, Import mapping, Validate mapping.

**States:** Missing mapping, inactive source structure, duplicate program code, unsupported stage, import partial failure.

**Backend data needed:** Academic structure, mapping records, validation results.

### 5.4 iEMIS Readiness

**Purpose:** Prepare school-level data for iEMIS-like reporting.

**Layout:** Checklist cards by institution, class/section, student profile, guardian, enrollment, documents, reports.

**Header actions:** Validate, Generate export, Download latest export.

**Backend data needed:** iEMIS readiness checklist, unresolved errors, report run history.

**Boundary:** This screen can link to M1/M4 edit screens but must not silently mutate student data.

### 5.5 UGC / HEMIS Readiness

**Purpose:** Prepare higher education data for UGC/HEMIS reporting.

**Layout:** Program summary, student statistics, staff statistics, scholarship summary, validation table, report preview panel.

**Header actions:** Validate HEMIS, Generate HEMIS export, Mark submitted.

**Confirmations:** Mark submitted requires date, method/reference, and reason/note.

**Backend data needed:** HEMIS summary, validation errors, export file, submission history.

### 5.6 QAA Readiness

**Purpose:** Track QAA status, expiry, and evidence package.

**Layout:** Status card, expiry timeline, evidence checklist, evidence file table, renewal notes.

**Header actions:** Add evidence, Generate QAA package, Create reminder.

**Backend data needed:** QAA profile, evidence list, readiness checklist, reminder policy.

### 5.7 Report History

**Purpose:** Preserve an auditable record of generated, exported, submitted, rejected, and archived report runs.

**Layout:** Paginated table with filters by report type, status, year, generator, submission state.

**Header actions:** New report, Export history.

**Protected files:** Download buttons must use authenticated File Registry helpers.

**Backend data needed:** Paginated report runs, file IDs, audit metadata.

## 7. State Matrix

| State | UI behavior |
|---|---|
| No profile | Empty state with setup action. |
| Missing required fields | Show blocking ERROR rows and fix links. |
| Validation queued | Disable generate action and show queued status. |
| Validation failed | Show errors grouped by entity and severity. |
| Ready | Enable report generation. |
| Export queued | Show progress/queued state, no fake download. |
| Export failed | Show retry where permission allows. |
| Submitted | Lock submission marker unless correction permission exists. |
| Rejected | Show rejection reason and regeneration action. |
| File unavailable | Show protected-file unavailable state. |

## 8. Security and Authorization

1. Tenant scope is mandatory on every query and mutation.
2. Parent/student users must not access M15 admin screens.
3. Report exports must be protected files, not raw storage URLs.
4. Sensitive demographics must be permission-filtered and purpose-limited.
5. Platform support override requires reason and audit.
6. Mark-submitted and report archive actions require permission and audit.
7. Dashboard aggregates must be server-owned.

---

## 9. Nepal-Specific Requirements

1. Support BS/AD display where backend provides both dates.
2. Support Nepal province/district/municipality/ward fields.
3. Keep iEMIS and UGC/HEMIS separate.
4. Support Nepali and English institution names.
5. Support QAA accreditation metadata.
6. Support PAN/VAT fields through shared institution profile but leave billing behavior to M3B.
7. Do not collect sensitive category data unless reporting need and tenant policy allow it.

---

## 10. Done Definition

M15 frontend is ready only when:

1. Every screen uses real backend/OpenAPI contracts.
2. No official summary is calculated in browser.
3. Validation errors come from backend.
4. Export generation is server-owned.
5. File download uses File Registry helpers.
6. Permissions are enforced in route guards and backend.
7. Empty/error/locked/permission states exist.
8. Submission history is auditable.
9. iEMIS and UGC/HEMIS remain clearly separated.
10. The UI does not claim official submission or integration without backend evidence.
