1. System Architecture & Overview
SchoolOS is a multi-tenant SaaS platform built on a NestJS monorepo architecture. Each school is an isolated tenant. All 10 modules share a single PostgreSQL database scoped by school_id. The platform is deployed as a single API with modular activation per subscription tier.

2. Module 1 — Admissions & Student Profile Management
ARCH
M1 is the root of the entire SchoolOS data model. Every other module — attendance, fees, marks, activity feed — queries the student record. Data quality here determines data quality everywhere.

The Admissions module manages the complete lifecycle of a student's relationship with the school — from initial enrollment, through annual class promotions, to graduation or transfer exit. It serves as the single source of truth for all student identity, placement, and guardian data used by all other modules.

2.4 Key Business Rules
	•	Student ID format is SCH-YYYY-NNNN. YYYY = academic year start. NNNN = sequential per school, zero-padded. Once assigned, immutable.
	•	Sibling linking: admin manually links 2+ students from same family. Module 3 reads this link to apply sibling discounts automatically.
	•	Duplicate detection: system checks same name + same DOB before final enrollment. Admin must confirm intentional duplicate.
	•	A student record cannot be deleted while they have an outstanding fee balance — must be cleared or waived first.
	•	Disability flag is a required iEMIS field — system prompts admin to confirm 'No known disability' rather than leaving blank.
	•	Privacy Act 2075: guardian consent timestamp recorded on first app login. No marketing communications until consent recorded.
	•	Alumni archiving: passed-out students moved to alumni status. Records retained 12 months, then eligible for deletion on request.

2.7 Edge Cases & Validation Rules
	•	Roll number conflicts: if manually set roll number already exists in that class-section-year, system blocks and shows conflict.
	•	Transfer students: previous school records stored but not validated. Admin manually confirms class placement.
	•	Bulk CSV import: each row validated individually. Invalid rows rejected with specific error messages in downloadable error report.
	•	Photo storage: images compressed on upload to max 800px wide, 80% quality. Signed URLs expire in 1 hour.

3. Module 2 — Attendance Management
CRITICAL
THE 3-TAP RULE: Open class → tap absent students (all pre-marked present) → submit. If attendance takes more than 30 seconds, teachers stop using it within 2 weeks. This is the #1 UX constraint for the entire module.
 
3.1 Purpose & Scope
The Attendance module handles daily marking of student and staff attendance, generates analytics for promotion eligibility and defaulter management, feeds leave data into payroll, and triggers real-time parent notifications. The module must function fully offline with sync-on-reconnect.

3.3 Offline Operation Architecture
	•	Attendance data saved to device SQLite database on submission.
	•	Network check runs every 30 seconds. On reconnection, unsynced records queued and uploaded in order.
	•	Submission timestamp = device time at the moment teacher tapped Submit — not the server receipt time.
	•	Conflict resolution: if the same class-date attendance is submitted twice (two devices), system uses the most recent submission timestamp and flags conflict for admin review.
	•	Submission lock: attendance locked 6 hours after submission. Admin override required to modify after lock.

13. Authentication, Authorization & Security Architecture
ARCH
SchoolOS uses a plug-and-play NestJS auth library (@corp/auth) built on custom JWT + passport-jwt with full RBAC, audit logging, and multi-tenancy. Zero auth code needed in individual modules — all protection is applied globally via AuthModule.forRoot().
 
13.1 Authentication Overview
Authentication in SchoolOS is handled by the @corp/auth library imported by the school-os NestJS app. The library provides complete JWT-based stateless authentication with refresh token rotation stored in Redis, bcrypt password hashing (cost 12), OTP delivery via SMS, and brute-force rate limiting.

13.4 JWT Token Strategy
Access Token
	•	Algorithm: HS256 (HMAC SHA-256)
	•	Expiry: 24 hours for admin/teacher web app — short enough to limit exposure if stolen
	•	Payload: { sub: userId, tenantId: schoolId, role: 'accountant', permissions: [...], iat, exp }
	•	Storage: Stored in memory (React state) or httpOnly cookie — never localStorage
 
Refresh Token
	•	Expiry: 30 days for parent mobile app — longer to reduce friction for daily use
	•	Storage: Stored in Redis with key pattern refresh:{userId}:{tokenFamily}
	•	Rotation: Every refresh call issues a new refresh token and invalidates the old one (token family pattern)
	•	Revocation: Admin can invalidate all sessions for a user by deleting their Redis keys

13.8 Guard & Decorator Patterns (NestJS)
Guard Execution Order
On every protected request: JwtAuthGuard → RolesGuard → PermissionsGuard. A request must pass all three guards before reaching the controller handler.
 
Guard / Decorator
Purpose
Usage Example
JwtAuthGuard
Validates JWT access token on every request. Attaches decoded user to request object.
@UseGuards(JwtAuthGuard) on controllers or set globally
RolesGuard
Checks if the authenticated user's role matches the @Roles() decorator on the handler.
@Roles('principal', 'accountant') on a controller method
PermissionsGuard
Granular permission check beyond roles — checks specific action strings.
@Permissions('fees:collect', 'fees:waive') on a handler
@Public()
Decorator that bypasses JwtAuthGuard for public endpoints.
@Public() on POST /auth/login, POST /auth/otp/request
@CurrentUser()
Injects the authenticated user object into handler parameters.
handler(@CurrentUser() user: User) { ... }
AuditInterceptor
Logs every write operation: actor, action, entity, old value, new value, IP, timestamp.
Applied globally — every POST/PATCH/DELETE is audited automatically
 
13.9 Multi-Tenancy Isolation
SECURITY
Multi-tenant isolation is enforced at the database query layer. All queries are automatically scoped by tenantId (school_id). Cross-school data access is architecturally impossible — not just policy, but enforced in the ORM.
 
	•	Every Prisma query includes an implicit WHERE school_id = :tenantId condition injected by a Prisma middleware.
	•	tenantId is extracted from the verified JWT payload — it cannot be spoofed by the client.
	•	Super Admin has a special bypass mechanism that requires an explicit tenant override header — logged in the audit trail.
	•	No direct database access for any frontend role — all data access goes through the NestJS API layer.
